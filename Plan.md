# FSFM DiFF Improvement Plan

## 1. Problem setting

We have decided to use **DiFF** as the final evaluation benchmark. The downstream recipe is:

- train on `FF++_DeepFakes(c23)`
- validate on `DiFF/val_subsets`
- test on `DiFF/test_subsets`

The original FSFM backbone is strong, but its standard fine-tuning recipe still has two clear weaknesses on DiFF:

1. **Global-head bias**: a single CLS-based classifier tends to compress all evidence into one global decision, which is weak for diffusion forgeries whose artifacts are sparse and local.
2. **Frequency blindness**: many DiFF samples are semantically plausible, so the most informative traces are often high-frequency inconsistencies, texture noise, and edge irregularities rather than large semantic boundaries.

There is also a third practical issue:

3. **Subset shift inside DiFF**: T2I, I2I, FS, and FE are generated by different pipelines, so a static classifier can overfit to one subset and fail to generalize to the others.

## 2. Chosen direction: FTF with Residual-View Consistency

We keep the current **Forgery-Aware Token Fusion (FTF)** idea, but make it stronger for DiFF by adding a complementary **Residual-View Consistency (RVC)** branch.

The final method name for the paper and code should be:

- **FTF-RVC** = Forgery-Aware Token Fusion + Residual-View Consistency

Core intuition:

- **FTF** mines suspicious patch tokens from the pretrained FSFM backbone, so the model does not rely only on one global CLS token.
- **RVC** feeds a high-pass residual view of the same image through the same detector, forcing the classifier to remain stable when the image is transformed into a frequency-focused representation.
- The combination is stronger than FTF alone because DiFF artifacts are not only local, but also subtle in the frequency domain.

This keeps the method lightweight:

- no extra annotations
- no face parsing masks
- no segmentation supervision
- no change to the basic binary classification objective

## 3. Method design

### 3.1 Spatial token fusion

For an input face image, the ViT backbone outputs:

- `c` = class token
- `p_1 ... p_N` = patch tokens

FTF computes a patch anomaly score using the class token as a reference:

```text
a_i = 1 - cos(p_i, c)
```

The most anomalous patches are selected with top-k pooling and softly aggregated:

```text
S = TopK(a_1 ... a_N, k)
l = sum_{i in S} softmax(a_i / tau) * p_i
```

where `l` is the local forgery descriptor and `tau` is the softmax temperature.

The global descriptor `g` comes from the original FSFM fine-tuning branch. We fuse them with a small gate:

```text
alpha = sigmoid(MLP([g ; l]))
f = alpha * g + (1 - alpha) * l
logits = Head(f)
```

### 3.2 Residual-view consistency

DiFF forgeries often leave weak high-frequency traces. To make the detector sensitive to them, we generate a residual view:

```text
x_r = H(x)
```

where `H` is a fixed high-pass filter, such as a Laplacian filter applied per channel.

During training, we run the same model on both views:

```text
z  = f_theta(x)
zr = f_theta(x_r)
```

and add a symmetric consistency penalty:

```text
L_rvc = 0.5 * [ KL(p || q) + KL(q || p) ]
```

with

```text
p = softmax(z / T)
q = softmax(zr / T)
```

The final objective is:

```text
L = L_ce(z, y) + lambda * L_rvc
```

This is more suitable for DiFF than FTF alone, because the residual branch explicitly amplifies the weak texture and edge traces that are easy to miss in the original RGB view.

### 3.3 Why this is better than FTF alone

FTF helps the model look at suspicious local tokens, but it still operates mainly in the spatial feature space. On DiFF, many failures come from frequency-level cues that are visually subtle but statistically informative. RVC complements FTF by:

- increasing sensitivity to high-frequency artifacts
- reducing dependence on global appearance shortcuts
- stabilizing predictions across the four DiFF subsets

In short:

- **FTF** fixes the "where to look" problem.
- **RVC** fixes the "what signal to trust" problem.

- add the local projection path and fusion gate

### 4.2 `main_finetune_DiFF.py`

Add / wire CLI arguments:

- `--token_fusion`
- `--topk_patches`
- `--fusion_tau`
- `--fusion_gate_hidden_dim`
- `--residual_consistency`
- `--rcr_lambda`
- `--rcr_temp`
- `--rcr_filter`

Behavior:

- if `--token_fusion` is set, instantiate `vit_base_patch16_tokenfusion`
- if `--residual_consistency` is set, enable the residual-view loss inside the training loop
- keep the pretrained checkpoint loading logic strict enough to catch accidental model mismatch

### 4.3 `main_test_DiFF.py`

Add the same model-selection arguments so evaluation can load the FTF checkpoint with the same architecture.

Notes:

- evaluation does not need the residual loss
- use the original-view path as the default test-time path for fair comparison

### 4.4 `engine_finetune.py`

Extend the training loop with:

- residual-view generation using a fixed high-pass filter
- a second forward pass on `x_r`
- symmetric KL consistency loss
- optional logging of the residual consistency term

The forward pass should still return logits only, so the rest of the codebase stays simple.

## 9. What to write in the paper

The paper should emphasize:

- DiFF is hard because diffusion forgeries are semantically plausible and the signal is sparse.
- A single global head is not enough, especially under subset shift.
- FTF helps the model focus on suspicious patch tokens.
- RVC makes the detector pay attention to high-frequency residual evidence and improves robustness across T2I, I2I, FS, and FE.
- The method is lightweight and does not require extra supervision.

We will collaborate directly on `main`:

- `main`: shared working branch for code changes, docs, and experiment scripts

### 10.1 Local developer workflow

Before starting any new edit session:

```bash
git checkout main
git pull --rebase origin main
```

When making changes:

```bash

```

If the other collaborator has pushed new commits before you finish:

```bash
git fetch origin
git pull --rebase origin main
```

### 10.2 Server sync workflow

On the server clone, always sync `main` before launching experiments:

```bash
git checkout main
git fetch origin --prune
git pull --rebase origin main
git status --short
```

Then run preprocessing or training from the synced commit.

### 10.3 Conflict rules

- never commit `datasets/` contents
- never commit checkpoints, logs, or tensorboard files
- never edit and push blindly from the server if the local developers have already moved `main`
- if local code and server code diverge, resolve conflicts locally, push the resolved `main`, then pull again on the server
- if a checkpoint is tied to a specific commit, record the commit hash in the experiment log
